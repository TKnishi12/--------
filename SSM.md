# ソフトシステムズ方法論（SSM）｜勉強会用メモ

## 0. このメモの主なインプット
- 2024年12月末「社会システムのシステムズアプローチ」講義
- ピーター・チェックランド著『ソフトシステムズ方法論』あああ

## 1. SSMの概要とメモ作成の背景
SSMは、関係者のあいだで「何が問題か／成功か／どこまで扱うか」が割れている状況（problem situation）を、対話とモデル化で“扱える形”にしていく方法論。

このメモは、2024年末の講義をきっかけに「仕事で直面している課題は『解く』以前に『何を問題とみなすか』を揃える必要がある」と感じたことを出発点に、SSMの要点を勉強会向けに整理したもの。

- SSMが効く状況：目的・評価指標・境界が争点になり、複数の世界観（W）が並立する
- 目指すもの：最適解よりも、共通理解と**合意可能な変化（accommodation）**（＝全員一致ではなく、動ける折り合い）

## 2. ハードアプローチとソフトアプローチ
- **ハード**：目的・指標・境界が明確 → 最適化／設計／計画に強い
- **ソフト**：目的や境界が争点 → 問題状況の構造化／合意形成／学習に強い

「ハード」と「ソフト」は対立概念というより、局面に応じて接続して使うものだと捉える方が実務的。社会と技術の変化に合わせて、それぞれの方法論が限界を経験し、補完しながら進化してきた「系譜」として見ると理解しやすい。

## 3. コア概念：役割・価値・規範
組織の問題が難しくなるのは、手段不足よりも**役割・価値・規範のズレ**が循環的に再生産されるため。SSMはこのズレを「厄介もの」ではなく、**構造化して扱う**。

- **役割**：誰が決め、誰が実行し、誰が責任を負うか
- **価値**：何を良い／成功とみなすか（優先順位）
- **規範**：「こうあるべき／許されない」という暗黙ルール

## 4. SSMの基本セット（7ステップ）
SSMの肝は「現実世界」と「システム思考」を往復して学ぶことにある。現実をそのままモデル化するのではなく、目的を持つ“仮想の活動システム”を一度組み立て（概念モデル）、現実との差分から改善案を引き出す。

1. 問題状況に入る（関係者・背景の把握）
2. 問題状況を表現する（リッチピクチャー等で外化）
3. 根底定義を作る（目的を持つ活動システムを言語化）
4. 概念モデルを作る（必要な活動を論理的に並べる）
5. 現実と比較する（ギャップ＝論点を洗い出す）
6. 実現可能で望ましい変化を検討する（文化・政治性も含めて折り合う）
7. 改善に向けて行動する（実行→学習→次へ）

### CATWOE
CATWOEは、根底定義（活動システムの定義）を詰めるための問いのセット。

- **C Customers**：影響を受ける人／受益者・被害者（誰が得をし、誰が困る？）
- **A Actors**：実行者（誰が実際にやる？）
- **T Transformation**：変換（何をインプットとして何に変える？）
- **W Weltanschauung**：世界観・正当化（なぜそれが重要で「良い」と言える？）
- **O Owners**：所有者・止める権限（誰が止められる／変えられる？）
- **E Environment**：外部制約（どんな制約の中でしかできない？）

（図2との接続）
- **役割**はA/Oに、**価値**はWに、**規範**はE（＋W）に現れやすい。

## 5. 学び・考察
現在、仕事では自社の業務全体（販売、物流、購買、会計、プロジェクトデリバリ等）をグローバルで標準化し、情報システムを刷新するプロジェクトに関わっている。業務領域や拠点（日本本社／海外拠点）の状況、利害が異なるため、次のような「割れ」が起きやすい。

- 何を問題とみなすか（優先課題の置き方が違う）
- 何を成功とみなすか（標準化・統制 vs 現地適合・柔軟性など）
- どこまでを対象に含めるか（プロセス境界／責任境界が曖昧）

結果として、納期圧が強い局面では、短期的に前に進む案（拠点固有業務を無視した設計、領域間のつながりを十分に考慮しない設計）に寄りやすい。SSMの観点では、こうした状況は「解く」以前に問題状況を構造化し、合意可能な変化（accommodation）をつくる必要がある局面だと捉えられる。SSMは、そのためのステップを明示している。
